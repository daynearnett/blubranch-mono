PART 0 — ORIENTATION FOR YOU, CLAUDE CODE
================================================================================

WHO YOU ARE
You are Claude Code, working with a non-technical-lead founder (the user's
business partner) to rebuild BluBranch from a v1 alpha into a v2 production
beta. The user (founder #1) has provided you mockups and this guide. Founder
#2 is your direct collaborator. He will run the commands, review your PRs,
and answer product questions when this guide is silent.

WHO IS NOT IN THE LOOP
Founder #1 (the one who made this guide) is busy with regulatory and business
work. Do not assume he is available for back-and-forth. This guide is his
voice. If this guide and Founder #2 disagree on something product-related,
TRUST THIS GUIDE and surface the conflict explicitly.

HOW TO ACT
- Default to shipping. This is a beta with two users (the founders). Done
  beats perfect. But "done" still means "no data corruption, no broken auth,
  no public security holes."
- Ask before destroying. If you would delete a table, drop a column, change
  an auth scheme, or rename anything user-facing, stop and confirm.
- Don't invent product decisions. If the mockup is ambiguous, pick the
  smallest interpretation and flag it as a TODO. Don't add features.
- Verify what the codebase actually is before you touch it. The user is not
  sure what state v1 is in. Read package.json, prisma/schema.prisma, the
  router setup, and the env file FIRST. Then decide whether to refactor in
  place, rewrite the frontend, or start fresh.
- Use the v3 ticket backlog (BluBranch_Tickets_v3.md, in the same folder as
  this guide) as the master to-do list. This guide explains WHY; the tickets
  enumerate WHAT.
- Write tests for anything that touches money, auth, or verification status.
  Everything else can ship with manual QA.

WHEN TO STOP AND ASK FOUNDER #2
- DB schema changes that aren't pure additions
- Anything involving payments, Stripe, or money flow
- Anything that sends real email or SMS to non-founder users
- Adding a third-party SDK or paid API
- Anything that would expose user data publicly (a robots-allowed page, an
  unauthenticated API endpoint, an embedded analytics pixel)

PART 1 — PRODUCT FOUNDATION
================================================================================

WHAT BLUBRANCH IS
A professional network for blue-collar tradespeople. Think "LinkedIn for
electricians, plumbers, HVAC techs, carpenters, welders, ironworkers, masons,
roofers, and the apprentices/foremen/contractors who hire them."

THE POSITIONING (memorize this; everything else follows)
"The professional network built for the Blue Collar."

It is NOT:
- A gig marketplace (Thumbtack, TaskRabbit) — those are consumer→worker
- A staffing app (Workrise, Trade Hounds) — those are temp-labor focused
- A general job board (Indeed, ZipRecruiter) — those have no profile/network
- LinkedIn — which is white-collar by tone, vocabulary, and prestige signals

It IS:
- A real professional identity for tradespeople (verified license, real work
  history, photos of actual work)
- A network they can grow with people they trust (crews, foremen, unions)
- A jobs board where employers compete for verified workers, not the other
  way around
- A trusted directory employers actually use because licenses are verified

THE TWO CORE USERS

  WORKER (90% of users, always free)
  - Trade professional, 18–60, mostly male, mostly mobile-first
  - Has a phone, has a trade, may or may not be in a union
  - Skeptical of social media but uses Facebook Groups, Instagram, YouTube
  - Doesn't want to fill out a resume. Wants employers to find HIM.
  - Cares about: pay, distance, day vs. night shift, union status, the crew
  - Doesn't care about: clout, follower counts, "thought leadership"

  EMPLOYER (the revenue side, paid)
  - Small-to-mid contractors, shop owners, service companies, GC schedulers
  - Posts jobs and searches the directory
  - Pricing tiers (see Part 4 — Monetization): $49 Basic / $129 Pro /
    $299/mo Unlimited
  - May also be a worker user; same person, two modes (handle gracefully)

TONE OF VOICE — CRITICAL
- Plain. Direct. Confident. No corporate-speak.
- "Find local trade jobs" ✓     not "Discover opportunities tailored for you"
- "Get verified. Get hired."  ✓ not "Unlock career advancement"
- "No fluff." ✓                 not "Streamlined workflows"
- Use trade vocabulary correctly: journeyman, apprentice, foreman, master,
  service call, conduit, panel, IBEW Local, etc. Wrong vocabulary will get
  the app rejected by trade users instantly.

WHAT WE'RE OPTIMIZING FOR (beta-stage priorities, in order)
  1. The founders can use it daily without rage-quitting
  2. A real Chicago electrician can sign up, get verified, and apply to one
     real job without help
  3. A real Chicago contractor can post a paid job and receive at least one
     application
  4. No PII or payment data leaks
  5. Looks good in a screen recording (for fundraising / partner outreach)

NOT priorities for v2:
  - Scale beyond ~1,000 users
  - SEO
  - Native apps (web-mobile-responsive is fine)
  - Internationalization

PART 3 — TECHNICAL FOUNDATION
================================================================================

STACK ASSUMPTIONS
You'll find out the actual stack when you read the v1 codebase, but here's
what we recommend if you have any latitude:

  Frontend:  Next.js 14+ (App Router) + TypeScript + Tailwind
             React Server Components where they make sense
             shadcn/ui for primitive components (then customize to match
             design system)
  Backend:   Next.js API routes OR a separate Node service — whichever the
             v1 uses, keep it
  Database:  PostgreSQL via Prisma. The v1 uses Prisma; keep it.
  Auth:      NextAuth.js with Email + (eventually) Google OAuth
             SMS via Twilio for phone verification
  Hosting:   Vercel for web, Railway or Render for the DB
  Storage:   S3 or Cloudflare R2 for photo uploads
  Payments:  Stripe for employer subscriptions (do NOT build until P0/P1
             work is done; ticket says P2)

ENV VARS (the codebase needs these; check what's actually configured)
  DATABASE_URL
  NEXTAUTH_SECRET
  NEXTAUTH_URL
  RESEND_API_KEY  (or SENDGRID_API_KEY) — for email verification
  TWILIO_ACCOUNT_SID, TWILIO_AUTH_TOKEN, TWILIO_PHONE_NUMBER — for SMS
  STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET — for employer payments (later)
  S3_BUCKET, S3_ACCESS_KEY, S3_SECRET_KEY, S3_REGION — for photos
  GEOCODER_API_KEY (Mapbox or similar) — for location autocomplete

DATA MODEL (target schema; v1 may differ — reconcile carefully)

  User
    id, email, emailVerified, passwordHash, phone, phoneVerified,
    firstName, lastName, createdAt, updatedAt

  Profile (1:1 with User; separate so we can soft-delete or hide profiles)
    id, userId, trade, tradeYears, currentCompany, currentTitle,
    headline, about, location (city), state, zip, lat, lng,
    avatarUrl, bannerUrl, openToWork (bool), profileCompleteness (computed)

  License
    id, userId, type (e.g. "IL Journeyman Electrician"), number, issuingState,
    expiresAt, verified (bool), verifiedAt, verifiedBy (system/manual)

  WorkPlace (verified employment, separate from currentCompany text)
    id, userId, companyName, role, startDate, endDate, current (bool),
    verified (bool), verifiedAt, verificationMethod (email/manual)

  Skill (many-to-many via UserSkill)
    id, name, trade (which trade it belongs to)

  Connection
    id, userIdA, userIdB, status (pending/accepted/rejected),
    requestedAt, respondedAt
    NOTE: enforce A < B at insert time to avoid duplicates

  Post
    id, userId, body, mediaUrls[], audience (anyone/connections),
    location, tradeTag, createdAt

  PostLike, PostComment, PostShare — standard

  Job (employer-posted)
    id, employerId (User with employer role), companyName,
    title, description, payMin, payMax, payUnit (hr/yr),
    location, lat, lng, jobType (FT/PT/contract), unionLocal,
    urgent (bool), promoted (bool), promotedUntil, status (draft/live/closed),
    createdAt, expiresAt

  Application
    id, jobId, userId, status (submitted/viewed/shortlisted/passed/hired),
    submittedAt
    NOTE: must require phoneVerified=true at submission time

  Conversation, Message — direct messaging between users

  EmployerSubscription
    id, userId, tier (basic/pro/unlimited), stripeSubscriptionId,
    status (active/canceled/past_due), currentPeriodEnd

CRITICAL DATA RULES
- Email and phone uniqueness is enforced at the DB level (UNIQUE constraint).
- Phone verification is required BEFORE a user can apply to a job, not
  before signup. v1 has this backwards (forces phone at signup, fails late).
- A "verified" user means: license verified OR workplace verified. Either
  alone qualifies for the green badge. Both is ideal.
- Don't soft-delete users; hard-delete on request (GDPR-friendly). Cascade
  to posts, applications, messages. Keep job postings (anonymize the
  employer reference).
- location should be normalized: never trust raw user input for searching.
  Use a geocoder, store lat/lng, search by radius.

API CONVENTIONS
- REST routes under /api/, kebab-case (e.g. /api/jobs/:id/apply)
- Always return JSON. Errors have shape: { error: { code, message } }
- Auth via NextAuth session cookie. Server actions can call getServerSession.
- Rate limit aggressively on: signup, login, password reset, application
  submit, message send. Use Upstash Redis or similar.
- All write endpoints require CSRF token (NextAuth handles for session-based)

PERFORMANCE TARGETS (beta, but enforce)
- Largest Contentful Paint on Feed: < 2.5s on 4G
- DB query budget per page: < 5 queries
- N+1 detection: install prisma-query-event-handler in dev
- Image uploads must be resized server-side (max 1600px wide, JPEG 80%)

PART 4 — GLOBAL BEHAVIORS (apply across many screens)
================================================================================

AUTH FLOW
- Email + password (NO username, NO phone-first signup)
- Email verification required before account is "active" (i.e. before
  bottom-nav appears). Until verified, user sees a single screen prompting
  re-entry of the 6-digit code.
- 6-digit codes expire in 10 min, max 5 attempts, resend rate-limited
  to 60s.
- Password rules: 6+ chars, must contain a digit OR symbol. (We picked
  permissive on purpose for trade users; pair with strong bcrypt + rate
  limiting + breach-password check via HIBP API.)
- Session: long-lived (30 days), refreshed on activity.

NAVIGATION
- Bottom nav is the source of truth for primary destinations:
    Feed | Network | Post(FAB, center) | Jobs | Me
- Top bar on primary screens: my avatar (left, links to /me), search pill
  (center, links to /search), messages icon (right, links to /messages)
- Back navigation everywhere except primary tab roots. Hardware back must
  work on mobile (don't trap user with nav-stack bugs).
- Settings (gear) lives inside profile pages, NOT in the top bar. Some
  mockups show a gear on top bars — those are wrong; treat as a profile
  link.

VERIFICATION SYSTEM (this is the killer feature; nail it)

  Two paths, EITHER one qualifies a user as "verified":

  Path A — License Verification
  - User enters license type + number + state
  - We check against state licensing board where API exists (IL, CA, NY,
    TX have public lookups). For others, manual queue (founder reviews
    within 24h in beta).
  - On approval: set license.verified=true and user.verified=true,
    send notification.

  Path B — Workplace Verification
  - User enters company email matching domain we've vetted
  - We send confirmation email to that address with one-click verify link
  - On click: set workplace.verified=true and user.verified=true.

  The green-circle-checkmark badge appears anywhere a verified user shows:
  - Avatar corner (their own profile, big)
  - Inline next to name (post bylines, search results, connection lists,
    messages, network suggestions)
  - DO NOT show the badge before verification completes. Pre-verification
    users get the avatar with no dot.

  If a user's license expires or workplace verification is revoked, the
  badge disappears immediately. Send an email and a banner notification.

OPEN TO WORK
- Off by default. Worker opts in from profile or settings.
- Toggle exposes user in employer searches and shows a small orange ring
  around avatar in some surfaces (NOT a full banner — keep it discreet).
- Audience selection: "Recruiters only" or "Everyone, including my current
  employer" — default to recruiters-only.

CONNECTIONS
- Symmetric (LinkedIn-style), not asymmetric (Twitter-style)
- 1st degree = direct connection
- 2nd degree = friend-of-a-friend
- 3rd+ = not shown (we only show 1st & 2nd in suggestions/searches)
- Invitation flow: request → notify → accept/dismiss. Dismissed requests
  don't notify the sender.
- Connection limit: 30,000 (LinkedIn-style; functionally infinite for our
  users)

MESSAGING
- Direct only (no group chats in v2)
- Employer messages are visually flagged with an "EMPLOYER" tag and an
  orange dot for unread
- Workers can mute employer messages
- No read-receipts in v2 (privacy default; revisit later)
- Rate limit: 50 messages/day per user to non-connections

EMPLOYER MODE
- An employer is a User with an EmployerSubscription
- Employer-only screens (job posting flow, applicant management) are not
  in the v2 mockups — those are tracked separately. Defer for now.
- An employer user still has a worker-side profile. Some are dual-role
  (a foreman who hires AND who is a worker). Build for this.

MONETIZATION (DO NOT BUILD UNTIL FOUNDERS GIVE GO-AHEAD)
- Workers: free forever, no upsells, no premium tier. This is the brand
  promise. Do not add upsells anywhere on worker surfaces, even "optional"
  ones.
- Employers:
    Basic        $49 / single job, 30 days
    Pro          $129 / single job, 30 days, promoted slot
    Unlimited    $299 / month, unlimited jobs + promoted
- Payment via Stripe Checkout (hosted page; do not handle card data
  ourselves).

LOCATION / GEO
- Default radius: 25 miles from user's home zip
- Always normalize via Mapbox/Google Geocoder. Store lat/lng, not strings.
- Distance shown on jobs board ("4.2 mi") is calculated client-side from
  user's stored lat/lng to job's lat/lng. Round to one decimal.

NOTIFICATIONS (defer beyond essentials)
- v2 must support: email verification, password reset, new connection,
  new message, application status change, license verification result
- Skip push notifications for v2 (web app, no native shell)
- Allow users to turn off non-essential email in /settings/notifications

ANALYTICS / TELEMETRY
- Use PostHog (self-hosted or cloud free tier) for product analytics
- DO NOT add Google Analytics. The audience is privacy-skeptical and we
  control no ad spend that would justify it.
- Capture: signup_complete, verify_complete, job_apply, job_post,
  message_send, connection_accept
- Strip PII from event properties.

ACCESSIBILITY
- WCAG 2.1 AA minimum
- All buttons have visible focus states (we removed default outline in
  design — restore it for :focus-visible)
- Form inputs have associated labels (not placeholder-only)
- Color contrast: text on orange MUST be white (passes); text on amber
  MUST be #412402 (passes)
- The verified badge is a visual signal; always pair with a `title` or
  `aria-label` of "Verified".

SCREEN 01 — Welcome Slide 1: Find Local Trade Jobs
--------------------------------------------------------------------------------
PURPOSE
Entry point for new users. Sells the jobs benefit on slide 1 of a 3-slide
carousel before signup.

HIDDEN CONTEXT
- Carousel auto-advances every 5s, but ALSO swipeable left/right and the
  dots are tappable. Pause auto-advance on user interaction.
- The "Skip" link goes straight to Screen 4 (signup name). It does NOT
  open a separate login flow — login is at the bottom of every slide
  ("Log in to existing account").
- The mockup PLUMBER card with $42-48/hr is decorative, not real data.
  Render as a static illustration, not a query.

BEYOND MOCKUP
- Need an "@/components/WelcomeCarousel" with auto-advance + swipe + dots
- Skip and Login states need to remember "intent" if user goes through
  signup then later wants to log in
- First-visit detection: cookie-based. Returning users skip the carousel
  on second visit and land directly on the login screen.

RISKS
- iOS Safari momentum-scroll can fight a carousel. Use a battle-tested
  library (Embla, Keen Slider) — DON'T hand-roll.
- Auto-advance is annoying if the user is reading. Pause on hover/touch.
- The illustration is SVG — make sure it doesn't break in dark mode if
  we add it later (it shouldn't; backgrounds are explicit colors).

DEPENDENCIES
- Routes to: Screen 4 (signup) or login screen (not in v2 mockups; basic
  email+password form is fine)

SCREEN 04 — Onboarding: Add Your Name
--------------------------------------------------------------------------------
PURPOSE
Step 1 of signup. Collect first + last name only.

HIDDEN CONTEXT
- Progress bar at 15% — we have ~6 logical steps total. Don't recalculate
  per screen; treat 15/30/45/60/85/100 as fixed waypoints.
- The X in the top-left dismisses signup ENTIRELY and returns to Screen 1.
  Show a confirm dialog: "Quit signup? Your progress will be lost."
- The back arrow in subsequent steps goes BACK one step (not all the way
  out).
- "Continue" button stays disabled until both fields have non-whitespace
  content of length ≥ 2.

BEYOND MOCKUP
- Name validation: strip leading/trailing whitespace, reject if contains
  digits or special chars beyond apostrophe and hyphen (O'Brien, Smith-
  Jones are valid; "John123" is not).
- Store progress in localStorage so a tab refresh doesn't kill signup.
  Clear the cache once signup is complete.
- We need an explicit "signup intent" record server-side once we have an
  email; until then, it's all client state.

RISKS
- Don't auto-capitalize. Some trade names are stylized. Let the user type
  what they want.
- Don't enforce a min length of 3+ — Asian-language names can be 2 chars.
- This screen has no PII server-side yet; nothing to leak. Good.

DEPENDENCIES
- Next: Screen 5 (email+password)

--------------------------------------------------------------------------------
SCREEN 05 — Onboarding: Email & Password
--------------------------------------------------------------------------------
PURPOSE
Step 2. Collect email + password and the terms-acceptance checkbox.

HIDDEN CONTEXT
- The Terms checkbox is shown PRE-CHECKED in the mockup. That is wrong
  for legal reasons. Default to unchecked. Button stays disabled until
  it's checked.
- "Send verification code" — yes, we send the code on submit of this
  screen, before the user fills anything else out. If signup is abandoned,
  we just have an unverified email in the DB; reaper job cleans up after
  72h.
- The eye/show-password icon (not shown in mockup) — add it. Trade users
  on construction sites with gloves will fat-finger passwords constantly.
- Email duplicate check happens HERE, not at the end. If the email is
  taken, show inline error "An account with this email already exists.
  Log in instead?" with a "Log in" link.

BEYOND MOCKUP
- Password strength meter (simple — Weak/OK/Strong based on length + char
  variety). Visual feedback only; do not block on it.
- HIBP (haveibeenpwned) API check on submit. If pwned, warn user but
  allow continue. This is a strong UX win at low cost.
- Terms and Privacy links open in-app modal, not new tab. Drafted versions
  should exist in /legal/terms.mdx and /legal/privacy.mdx; if they don't
  yet, create stubs and TODO them.

RISKS
- Don't store password client-side ever. Hash server-side with bcrypt
  (cost 12).
- The verification code generation is server-side. Don't generate it
  client-side and POST it.

- Step 2 of 2 is the actual phone-code-entry screen (similar to Screen 6
  but with SMS instead of email). Not in v2 mockups; build it analogously.
- If user is already phone-verified, skip both steps and apply directly,
  showing a brief confirmation modal: "Application sent to {Company}!"
- The "your profile" review section is read-only. Edits route the user
  out to profile editor; their apply intent is preserved.
- All checkmarks (Name, Trade, License, Email) are green because the
  user got here via a profile that already has these. If any is missing
  (rare edge case), show a yellow warning instead and block apply with
  "Add your license to apply."

BEYOND MOCKUP
- SMS provider: Twilio. Phone number normalization: store E.164 format
  (+15551234567). Validate via libphonenumber.
- After phone verify: application is submitted, user lands on a
  confirmation screen with "What happens next?" copy and a CTA to view
  more jobs.
- Application notifies employer via in-app and email. Email template:
  "{Name} applied to {Job}. View their profile here."

RISKS
- Twilio costs money per SMS. Rate-limit: 3 SMS per phone number per
  hour. Block re-verification of same phone < 24h apart.
- Phone uniqueness: a phone can be associated with one user at a time.
  If user 2 tries to verify a phone owned by user 1, block with "This
  phone is already in use."
- Application status visibility: workers see their applications in a
  list (separate ticket — not in v2 mockups). Employers see applicants
  in their dashboard (also separate).

DEPENDENCIES
- Twilio configured
- Phone uniqueness enforcement
- Application table + email notifications

--------------------------------------------------------------------------------
SCREEN 24 — Search Landing: Recent & Trending
--------------------------------------------------------------------------------
PURPOSE
The search experience entry point. Shows recent searches + suggestions.

HIDDEN CONTEXT
- Recent searches are user-specific, stored server-side. Last 10. User
  can clear individual or all.
- "Try searching for" pills are personalized by trade. An electrician
  sees electrician-relevant suggestions; a plumber sees plumber ones.
- "Trending in trades" pills are global (or city-scoped). Computed
  from most-frequent search terms in the last 7 days. Refreshed daily.
- The search input is focused immediately on screen mount with the
  on-screen keyboard up (mobile).

BEYOND MOCKUP
- Search-as-you-type: trigger search at 2 chars, debounce 200ms.
- Search history: pruned to 10 most recent per user. Drop oldest as new
  ones come in.

RISKS
- Trending data could be embarrassing if a niche/joke search trends.
  Build a moderation layer: blocklist + admin reviews top trends.
- Empty trending state: if no data, fall back to hard-coded popular
  searches.

DEPENDENCIES
- SearchLog table (user_id, query, timestamp)
- Trade-specific suggestion list

--------------------------------------------------------------------------------
SCREEN 25 — Search Results: Tabs & Filters
--------------------------------------------------------------------------------
PURPOSE
Results page after a search query is entered.

HIDDEN CONTEXT
- 4 tabs: People, Jobs, Companies, Posts. People is default. Tab counts
  in parens are live.
- Filter pills under tabs: "1st & 2nd" (degree), "Verified" only, by
  city, by union. These filters are tab-specific (People tab has degree
  filter; Jobs tab doesn't).
- Each result row in People shows connection degree (1st/2nd) and an
  action button (Message for 1st, Connect for 2nd/3rd).
- Companies tab (not shown in mockup detail) lists employer profiles —
  defer to v3.

BEYOND MOCKUP
- Pagination: 20 results per page, load more on scroll.
- Search infrastructure: use PostgreSQL full-text search for v2
  (tsvector). Elasticsearch/Algolia is overkill for our scale.
- Search query is logged (SearchLog table) for "Recent" surface.

RISKS
- Slow searches: index aggressively (name, trade, location, skills as a
  combined tsvector). Test with 10k seed users.
- Verified-only filter is a value moment — make sure it actually filters
  consistently. Test with mixed verified/unverified results.
- Don't expose 3rd-degree+ users in default search. Privacy.

DEPENDENCIES
- Full-text index
- Connection graph

SCREEN 26 — Settings: Account, Preferences, Sign Out
--------------------------------------------------------------------------------
PURPOSE
Settings hub.

HIDDEN CONTEXT
- Three sections: Account (profile, security, phone), Preferences
  (notifications, language), Support (help, feedback). Followed by Sign
  out (red text, centered).
- "Profile & visibility" routes to a profile editor for sections beyond
  the basics: about, headline, banner, etc.
- "Sign in & security" handles password change, 2FA (defer to v3), and
  active sessions ("you're logged in on Chrome (MacOS) and Safari
  (iPhone)").
- "Send feedback" should email founders directly (founder #1 and #2
  email) — easy to keep a finger on the pulse during beta.

BEYOND MOCKUP
- "Delete my account" — must be reachable from settings. Hidden under
  "Account" or "Sign in & security." Multi-step confirm + 7-day cooling
  period before actual deletion.
- Language: English only for v2. Show selector but only one option.
- App version (shown at bottom: "BluBranch v0.4.2 (beta)") — bump per
  release. Pull from package.json automatically.

RISKS
- Sign out must clear local storage + session cookie. Test cross-tab
  (signing out in one tab logs out in others). Use the BroadcastChannel
  API or storage events.
- Account deletion is GDPR-relevant. The 7-day cooling period must
  pause if the user cancels.
- Don't put dangerous links (delete account) close to safe ones (edit
  profile). Use visual separation + red color for destructive actions.

DEPENDENCIES
- Profile editor
- Account deletion job + audit log

SCREEN 27 — Messages: Inbox with Employer Filter
--------------------------------------------------------------------------------
PURPOSE
Direct-message inbox.

HIDDEN CONTEXT
- Top filter pills: All, Unread (3), Employers, My crew.
  - "Employers" filters to threads with users marked as having an active
    EmployerSubscription.
  - "My crew" is a v3 idea — defer for v2 (gray it out).
- Unread threads have the orange left-dot and a slightly tinted bg.
- Compose FAB (top right, orange) opens a new-message flow — pick a
  connection, type message, send.
- Tapping a thread opens the conversation view (not in v2 mockups —
  build a basic version: header with name/avatar + verified badge,
  scrollable messages, text input at bottom).

BEYOND MOCKUP
- Real-time updates: use polling every 30s for inbox, or WebSocket if
  the stack supports it (Next.js + Pusher or Ably is overkill — skip).
  Polling is fine.
- Unread counts in bottom nav: the msg icon shows a small orange dot
  when unread > 0 (some mockups show this).
- Notification settings: user can mute employer messages OR all
  non-connections.

RISKS
- Message spam from employers: rate-limit messages from employer
  accounts to non-connections (5/day per recipient).
- Block/report: per-thread three-dot menu. Block hides their messages,
  removes the connection, prevents future messages.
- Image attachments in messages: defer to v3. Text only in v2.

DEPENDENCIES
- Conversation + Message tables
- Connection table
- EmployerSubscription table

PART 6 — CROSS-CUTTING RISKS & FOCUS AREAS
================================================================================

LEGAL & COMPLIANCE
- Terms of Service + Privacy Policy MUST be live before any non-founder
  user signs up. Drafts at /legal/terms.mdx and /legal/privacy.mdx.
  Founder #1 will review wording (he's Series 66/7 and aware of comms
  rules — not legal counsel, but practical-enough).
- GDPR / CCPA: Build the data-export and data-delete endpoints from day
  one. Even if no European user signs up, US states are moving toward
  similar rules.
- Background check / I-9 data: we DO NOT collect this. Mention it in
  privacy policy to be clear.
- Equal Employment Opportunity: we DO NOT collect demographic data
  (race, gender, age). Trade is an occupational category, not
  demographic.

PAYMENTS
- Workers free forever. Hard product invariant. Do not introduce ANY
  paid worker feature without explicit founder sign-off.
- Employer payments: Stripe Checkout (hosted page). Never touch raw card
  data. Webhook handler must be idempotent.
- Refund policy: full refund within 24h of purchase if no applications
  received. After 24h, no refund. Document in TOS.

PERFORMANCE
- The slowest screens will be: Home Feed, Jobs Board, Search Results.
  These need DB indexes + careful query planning.
- Image-heavy screens (Portfolio, Posts): lazy-load, use srcset for
  responsive images, compress to JPEG-80.
- Mobile data usage matters — trade users are sometimes on metered
  cellular. Don't auto-play any video. Don't preload heavy assets.

ACCESSIBILITY
- Form labels (not placeholders) for every input.
- Focus visible on every interactive element.
- ARIA labels on icon-only buttons (search, message, gear, etc.).
- Color contrast ≥ 4.5:1 for body text. Test each combo we use.

ABUSE & MODERATION
- Reporting: every post, profile, job, and message has a "Report" affordance.
  Reports go to an admin queue (founder-managed in beta).
- Suspension: an admin flag (User.suspended=true) hides the user from
  all surfaces, blocks login. Use for proven bad actors.
- Verified-only protection: a verified user can be un-verified if proven
  fraud. Their badge drops immediately, profile flagged.

DATA RETENTION
- Unverified accounts: deleted after 72h of inactivity post-signup
  failure.
- Messages: retained indefinitely while accounts exist.
- Search logs: 90 days, then anonymized to aggregate trends.
- Logs (server): 30 days, then deleted.

SECURITY HARDENING (build these early)
- All forms POST with CSRF tokens
- All API endpoints check session
- Rate limit auth endpoints (5/min/IP)
- Bcrypt cost 12 for passwords
- HTTP Strict Transport Security
- Content-Security-Policy header (start permissive, tighten later)
- Sanitize all user input (DOMPurify on rendered HTML, parameterized
  queries on DB)

PART 7 — KNOWN BUGS FROM V1 TO FIX DURING REBUILD
================================================================================

These bugs exist in the v1 alpha. The rebuild MUST not reintroduce them.

P0 — Critical
- DB raw query failure: "column wp.location does not exist" / "column
  j.location does not exist". The schema is missing a column the query
  references. Likely cause: WorkPlace and Job tables need location/lat/
  lng columns added, OR queries need rewriting to use city+state.
  Decision: add location, lat, lng to both tables and use in all
  proximity queries.

- Phone duplicate check happens at END of signup as a popup, killing the
  user. FIX: phone is checked during apply-flow (Screen 23), not signup.
  Phone is NOT required at signup at all in v2.

- Email duplicate check is delayed. FIX: check during Screen 5
  submission and show inline error before sending verification code.

- Bottom nav uses iOS emoji icons. FIX: replace with Lucide SVG icons
  (Feed, Network, Plus, Jobs, User). Custom-styled per design system.

- No back navigation in onboarding. FIX: every step after Screen 4
  has a back arrow in the top bar; tapping returns to previous step
  WITHOUT losing data.

- No profile editing after creation. FIX: ship Screen 11 (verifications)
  AND a section-by-section profile editor accessible from Screen 12+.

- Welcome carousel doesn't exist in v1. FIX: ship Screens 1–3 first.

- Trade input is free-text in v1. FIX: autocomplete from curated list
  (Screen 8).

- iOS location popup is bypassed in v1. FIX: implement permission
  request flow (Screen 7).

- Employer onboarding is identical to worker onboarding in v1. FIX:
  defer — employer flow is out of v2 scope. But block path: don't ship
  ambiguous signup that lets employers in via the worker flow.

P1 — Critical
- Signup data doesn't flow to profile. Name/trade/company entered at
  signup must appear on the new profile (Screen 9) without re-entry.

- Enrichment cards don't exist. Build per Screen 10.

- Verifications page doesn't exist. Build per Screen 11.

- Verified badge is missing from most surfaces. Implement consistently
  per the global VerifiedBadge component.

- Profile edit doesn't exist. Build a section-by-section editor.

- Top bar isn't persistent. Build a sticky top bar per Part 4.

- Bottom nav doesn't match the new 5-tab structure (Feed/Network/Post/
  Jobs/Me). Rebuild per Part 4.

- Settings screen doesn't exist (sign out is buried in Portfolio/Posts).
  Build per Screen 26.

- Auto-headline isn't generated. Build the "{title} · {yrs} yrs ·
  {union}" formatter per Screen 12.

- Personalized skills don't exist. Skills are not trade-aware. Fix per
  Screens 8, 12.

- License verification doesn't exist. Build per Screen 11.

- Strong-match scoring doesn't exist. Build per Screen 21.

- Apply flow with phone gate doesn't exist. Build per Screens 22-23.

- Back nav in onboarding (already mentioned in P0 — confirm).

PART 9 — IF YOU GET STUCK
================================================================================

If you find yourself unsure of a design decision, here's the priority
order:

  1. Check this guide.
  2. Check BluBranch_Tickets_v3.md for explicit acceptance criteria.
  3. Check the v2 mockup PDF (BluBranch_Mockups_v2.pdf).
  4. Ask Founder #2 directly (he's running the build).
  5. Match LinkedIn's behavior — they've solved most professional-
     network UX patterns. If we're not deliberately different, default
     to their pattern (back nav, conversation lists, profile sections).
  6. Pick the simplest interpretation and flag it as TODO.

If you find yourself unsure of a TECHNICAL decision:
  1. Use the simplest, most-boring tool that works. Postgres > Redis-
     for-everything. REST > GraphQL. Server-rendered > SPA where simpler.
  2. Default to security over convenience.
  3. Default to data integrity over speed. We can optimize later.

If you find yourself adding a feature not in this guide or in the
tickets: STOP. Ask Founder #2. Scope creep kills 8-week builds.
